21.4 Creating Publication-Quality Figures

21.4.1 Concentration-Time Profile (Standard PK)

Code 
library(rxode2)

# Generate example data for figure
mod_fig <- rxode2({
  d/dt(depot)   <- -ka * depot
  d/dt(central) <-  ka * depot - (CL/V) * central
  conc <- central / V
})

set.seed(42)
n_fig <- 8
fig_times <- seq(0, 24, by = 0.2)

fig_data <- lapply(1:n_fig, function(i) {
  params <- c(ka = 1.0 * exp(rnorm(1, 0, 0.3)),
              CL = 5.0 * exp(rnorm(1, 0, 0.25)),
              V  = 50 * exp(rnorm(1, 0, 0.15)))
  event <- et(time = 0, amt = 500)
  sim <- rxSolve(mod_fig, params, event %>% et(fig_times))
  as.data.frame(sim) %>% mutate(ID = factor(i))
}) %>% bind_rows()

# Summary statistics
fig_summary <- fig_data %>%
  group_by(time) %>%
  summarise(
    Mean = mean(conc),
    SD = sd(conc),
    Median = median(conc),
    .groups = "drop"
  )

# Publication-quality figure
ggplot() +
  # Individual profiles (semi-transparent)
  geom_line(data = fig_data, aes(time, conc, group = ID),
            color = "gray70", alpha = 0.4, linewidth = 0.5) +
  # Mean ± SD ribbon
  geom_ribbon(data = fig_summary,
              aes(time, ymin = Mean - SD, ymax = Mean + SD),
              fill = "#0B6E6B", alpha = 0.15) +
  # Mean line
  geom_line(data = fig_summary, aes(time, Mean),
            color = "#0B6E6B", linewidth = 1.2) +
  # Median line
  geom_line(data = fig_summary, aes(time, Median),
            color = "#C44536", linewidth = 1, linetype = "dashed") +
  scale_y_log10(
    breaks = c(0.1, 0.3, 1, 3, 10, 30),
    labels = c("0.1", "0.3", "1", "3", "10", "30")
  ) +
  labs(
    title = "Mean (±SD) Plasma Concentration-Time Profile",
    subtitle = "Semi-logarithmic scale; n = 8 subjects",
    x = "Time (hours)",
    y = expression("Concentration (mg/L)"),
    caption = "Solid line: arithmetic mean; Shaded band: ±1 SD; Dashed: median"
  ) +
  theme_minimal(base_size = 12) +
  theme(
    plot.title = element_text(face = "bold"),
    plot.caption = element_text(color = "gray50", size = 9),
    panel.grid.minor = element_blank()
  )

21.8 Building a Reusable Code Library

Code 
# Template for a reusable PK function library
# Save as R/pk_functions.R

#' Calculate elimination rate constant from two concentration-time points
#'
#' @param c1 Concentration at time 1 (mg/L)
#' @param c2 Concentration at time 2 (mg/L)
#' @param t1 Time 1 (hours)
#' @param t2 Time 2 (hours)
#' @return Elimination rate constant (1/h)
#' @export
calc_ke <- function(c1, c2, t1, t2) {
  if (c2 >= c1) stop("c2 must be less than c1 (declining phase)")
  if (t2 <= t1) stop("t2 must be greater than t1")
  (log(c1) - log(c2)) / (t2 - t1)
}

#' Calculate half-life from ke
#'
#' @param ke Elimination rate constant (1/h)
#' @return Half-life (hours)
#' @export
calc_half_life <- function(ke) {
  log(2) / ke
}

#' Calculate accumulation factor
#'
#' @param ke Elimination rate constant (1/h)
#' @param tau Dosing interval (hours)
#' @return Accumulation factor
#' @export
calc_accumulation <- function(ke, tau) {
  1 / (1 - exp(-ke * tau))
}

# Example usage
# ke_est <- calc_ke(8.5, 2.1, 2, 8)
# t12 <- calc_half_life(ke_est)

21.9 Regulatory Considerations

21.9.1 FDA/EMA Expectations for PK/PD Analyses

  • Data standards: SDTM/ADaM formats in submissions
  • Software: R is accepted; document version and packages
  • Code submission: Increasingly expected for reproducibility
  • Model documentation: Complete model code, decision rationale
  • Sensitivity analyses: Demonstrate robustness of conclusions
  • Define.xml: Metadata for all datasets and variables

21.9.2 Exercises

  1. Create a Quarto template for a standard PK report with the sections listed above.
  2. Generate a publication-quality parameter table with estimates, RSE%, and 95% CI.
  3. Write a function library with utility functions for PK calculations (ke, half-life, AUC, accumulation).
  4. Document a complete PK analysis workflow with commented R code that a colleague could reproduce.

21.9.3 Common Mistakes

  • Not documenting data exclusions: Every excluded data point needs justification
  • Reporting only point estimates: Always include uncertainty measures
  • Over-polishing figures before model is final: Iterate on exploratory plots first
  • Not version-controlling analysis scripts: Git + meaningful commit messages
  • Assuming readers understand your model: Write for a clinical audience, not just modelers

21.9.4 Key Takeaway

A good PK/PD analysis is only as valuable as its communication. Structure your reports clearly, quantify uncertainty honestly, and make every analysis reproducible. The best model is useless if nobody understands what it means for patients.
